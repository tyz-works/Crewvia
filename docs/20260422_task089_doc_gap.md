## 3. アーキテクチャ設計の矛盾（Geordi への引き継ぎ事項）

以下は Phase B で Geordi が判断・解消すべき設計上の矛盾。ドキュメントより先に実装が決まらないと Phase E の執筆ができない。

### A-01: ntfy 送信の責務（最重要）

| 比較項目 | Lin の設計書 (`ntfy-approval-design.md`) | task_089.md Phase B 方針 |
|---|---|---|
| ntfy 送信者 | crewvia が直接 ntfy_publish | Taskvia が送信 (crewvia は `notify:true` を渡すだけ) |
| Taskvia レスポンス | `{ id, approve_url, deny_url }` が必要 | `{ id }` のみで十分 |
| crewvia の lib_approval_channel.sh | `ntfy_publish()` 関数を保持 | 関数削除可 |

**現在の実装**: pre-tool-use.sh が `ntfy: true` を Taskvia に送信しつつ、さらに `ntfy_publish()` も直接呼んでいる → **二重送信リスク**

**Troi コメント**: Phase B の実装方針が確定するまで Phase E のドキュメントは仮草稿止まり。

### A-02: `notify` フラグ vs 常時 publish

現状の `POST /api/request/route.ts`:
```typescript
// notify フラグに関係なく常に publishApprovalRequest を呼んでいる
await publishApprovalRequest(id, card.agent ?? "Unknown", card.tool, `承認待ち: ${card.tool}`);
return Response.json({ id });
```

提案書要件: `notify === true` 時のみ publish。未指定時は従来動作（publish しない）。

→ Phase B で Geordi が修正する必要あり。

## 4. TTL 整合性（情報共有）

Lin の設計書 §6 で既出。確認のため再掲:

| 対象 | 現在の TTL |
|---|---|
| approval カード (`approval:{id}`) | 600 秒 |
| ntfy トークン (`approval_token:{t}`) | 900 秒 (デフォルト) |
| pre-tool-use.sh ポーリング上限 | 600 秒 |

カード TTL(600s) < トークン TTL(900s) のため、ユーザーがタップした時点でカードが消えている可能性がある（稀だが）。Phase B で Geordi に確認を促すこと。
