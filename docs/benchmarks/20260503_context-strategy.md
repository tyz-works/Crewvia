# M-CTX-1: Worker コンテキスト戦略 比較分析レポート

**作成日**: 2026-05-03

## 1. 概要

Worker の Claude Code セッションにおけるコンテキスト管理の 3 戦略を、同一ベンチマークタスク 5 件で実測し、トークン消費・処理時間・コスト・スループットを比較した。

**結論サマリー**:
- **小タスク（≤5分/件）**: Strategy A（連続セッション）が最安値・最速に近く推奨
- **大タスク（>5分/件）**: Strategy B（/clear）でコンパクション回避とコスト抑制のバランスをとる
- **Strategy C（再起動）**: オーバーヘッドが顕著（+65%）で、現時点では非推奨

## 3. 集計結果

### 3.1 トークン消費比較

| メトリクス | Strategy A | Strategy B | Strategy C |
|-----------|-----------|-----------|-----------|
| **input_tokens** | 1,378 | 96 | 107 |
| **cache_creation_tokens** | 76,837 | 236,127 | 183,927 |
| **cache_read_tokens** | 2,275,532 | 2,285,088 | 2,823,670 |
| **output_tokens** | 10,430 | 14,448 | 18,139 |
| **総トークン** (合計) | **2,364,177** | **2,535,759** | **3,025,843** |
| **セッション数** | 1 | 3 | 4 |

#### 内訳注記

- **input_tokens**: A が最多（1,378）は最初のタスクで大量の新規入力（1,318 tokens）が発生したため。B/C は各セッション先頭で小量の入力のみ。
- **cache_creation_tokens**: B が最多（236K）。セッション再起動のたびにキャッシュコールドスタートが発生し、システムプロンプト・ツール定義を再キャッシュする。A は 1 回のキャッシュ構築（76K）で全タスクを賄う。
- **cache_read_tokens**: C が最多（2.82M）。各セッション起動時に前回キャッシュ全体を読み込むため。A/B は類似（2.28M）。
- **output_tokens**: C が最多（18K）。フレッシュなコンテキストで毎回自己説明的な出力が増える傾向。

#### Strategy A: タスク別キャッシュ推移（コンテキスト蓄積の効果）

| タスク | cache_creation | cache_read | wall_clock |
|-------|---------------|-----------|-----------|
| 1 (seg1) | 63,502 | 705,897 | 75.94s |
| 2 (seg2) | 5,985 | 499,400 | 50.55s |
| 3 (seg3) | 3,337 | 455,304 | 39.59s |
| 4 (seg4) | 1,559 | 215,452 | 24.21s |
| 5 (seg5) | 2,454 | 399,479 | 27.93s |

→ 最初のタスクで 63K tokens のキャッシュ構築後、後続タスクは 1.5〜6K のインクリメンタル追加に留まる。**単一セッションによるキャッシュ warm-up 効果が顕著**。

## 4. タスクサイズ × 戦略 クロス分析

今回のベンチマークは「小タスク（各 24〜76s、easy 中心）」に特化している。タスクサイズが変わると戦略の有効性は変化する。

| タスクサイズ | コンテキスト蓄積量 | A の問題 | B の有効性 | C の有効性 |
|------------|---------------|---------|-----------|-----------|
| **小** (〜5 min/task) | 低 | ほぼなし | 過剰（起動コスト大） | 過剰（再起動コスト大） |
| **中** (5〜20 min/task) | 中程度 | コンテキスト肥大化開始 | 有効（適度なリセット） | やや有効（オーバーヘッド許容範囲内） |
| **大** (>20 min/task) | 高 | コンパクション発生リスク大 | 重要（定期リセット必須） | 有効だが I/O コスト高 |
| **連続ロングセッション** (>1h) | 非常に高 | コンパクション不可避 | 必須 | 代替手段 |

**小タスクでの B/C のオーバーヘッド内訳（B の例）**:
- 3 セッション × 起動オーバーヘッド平均 ≈ 21s/session
- cache_creation 増加分: +159K tokens → +$0.597（A 比）
- 合計コスト増: +$0.657（+58%）

## 5. 推奨戦略（タスクサイズ別）

| タスクサイズ目安 | 推奨戦略 | 理由 |
|--------------|---------|------|
| easy / ≤5 min/task | **A（連続セッション）** | キャッシュ効果で後続タスクが高速化、コスト最小 |
| medium / 5〜20 min/task | **A → 自動 /compact（M6 hook）** | コンテキスト圧迫時のみ自動対処、手動リセット不要 |
| large / 20〜60 min/task | **B（/clear）** | コンパクション発生前に定期リセット、コンテキスト汚染防止 |
| ロングランニング / >1h | **B または C** | B: セッション継続でコスト抑制、C: 完全クリーンスタートが必要な場合 |

**推奨優先度**: A > B > C

## 6. ハイブリッド戦略の閾値提案

### 戦略 A → B 切り替え閾値

```
if context_tokens > THRESHOLD_CLEAR:
    invoke /clear  # Strategy B mode
elif compaction_count > 0:
    invoke /clear  # 強制リセット
else:
    continue  # Strategy A mode
```

推奨閾値:
- **THRESHOLD_CLEAR**: `100,000 tokens`（キャッシュ構築量が単一タスク分の 1.3× を超えた場合）
- 根拠: Strategy A では 1 タスク目の cache_creation が 63,502。その 1.6× = 約 100K を超えると後続タスクでのキャッシュ再利用率が低下する

### 戦略 B → C 切り替え閾値

```
if session_restart_count > 5 and avg_task_time > 300:
    switch_to_strategy_C  # 明示的な再起動の方が確実
```

推奨閾値:
- **avg_task_time > 300s** (5 分超/タスク) かつ **context_tokens > 500,000**

### 動的ハイブリッドの実装イメージ

```yaml
# config/crewvia.yaml (提案)
context_strategy:
  default: A
  thresholds:
    clear_on_context_tokens: 100000    # B に切り替え
    restart_on_context_tokens: 500000  # C に切り替え
    clear_on_task_duration_s: 300      # B に切り替え
  fallback_on_compaction: true         # コンパクション発生 → 即 B
```

## 7. PreCompact Hook (M6) との統合可能性

### M6 とは

`PreCompact` フック（M6）は Claude Code のコンテキスト圧縮（`/compact`）発動直前に呼び出されるフック。コンパクション前処理（ログ保存、チェックポイント、通知など）を自動化できる。

### 統合アーキテクチャ案

```
Strategy A (連続セッション)
      ↓
コンテキスト肥大化を検知
      ↓
PreCompact hook (M6) が発火
      ↓
[現状保存] task 完了状態・中間成果物を registry に保存
      ↓
Claude Code が /compact を実行
      ↓
圧縮後のコンテキストで作業継続
```

### M6 統合による Strategy A の強化

| 課題 | M6 なし（現状） | M6 統合後 |
|------|-------------|---------|
| コンパクション発生タイミング | 予測不能 | フック前にチェックポイント保存 |
| タスク途中のコンパクション | 作業状態消失リスク | タスク結果を自動 persist |
| 戦略切り替え判断 | 手動 | コンテキスト量に基づき自動 /clear or /compact を選択 |

### 実装優先度

1. **高優先度**: PreCompact フックでタスク中間状態を `queue/missions/*/tasks/tNNN.md` に保存
2. **中優先度**: コンパクション発生時に Director に通知（Taskvia 経由）
3. **低優先度**: コンパクション閾値に近づいたら事前警告（PreToolUse フックで検知）

## 8. クォータ窓 5h での完走数試算

### Wall Clock ベース試算（API 制限なしの上限）

| 戦略 | 5 タスク当たり実経過時間 | 5h (18,000s) でのタスク数 |
|------|------------------|----------------------|
| **A** | 239s | **377 タスク** (75 バッチ × 5) |
| **B** | 276s | **326 タスク** (65 バッチ × 5) |
| **C** | 394s | **228 タスク** (45 バッチ × 5) |

### トークンレート試算

| 戦略 | 総トークン (5 tasks) | 実経過時間 | tokens/min |
|------|-------------------|---------|-----------|
| A | 2,364,177 | 239s | **593,477** |
| B | 2,535,759 | 276s | **551,472** |
| C | 3,025,843 | 394s | **460,583** |

### API 制限を考慮した試算

Claude Sonnet 4.6 の典型的 TPM 制限を 800k tokens/min（Production Tier）と仮定した場合:

- 5h バジェット: 300 min × 800k = **240M tokens**
- A: 240M / (2,364,177/5) = **507 タスク**（制限内で wall clock が先にボトルネック）
- B: 240M / (2,535,759/5) = **473 タスク**
- C: 240M / (3,025,843/5) = **397 タスク**

→ 今回の小タスク規模では **API TPM 制限よりも wall clock レイテンシがボトルネック**。  
→ 5h 窓での最大完走数は Strategy A で **最大 377 タスク**（wall clock 制約）。

### 実運用補正係数

- タスク割り当て遅延（plan.sh pull 待機）: -10〜20%
- Worker 起動オーバーヘッド（初回）: 固定 10s/session
- 保守的見積もり（A 戦略）: **300〜340 タスク / 5h**

## 9. 結論

### 総合評価

| 評価軸 | Strategy A | Strategy B | Strategy C |
|-------|-----------|-----------|-----------|
| コスト | ★★★ | ★★ | ★★ |
| 速度（小タスク）| ★★★ | ★★ | ★ |
| コンテキスト汚染耐性 | ★ | ★★★ | ★★★ |
| セットアップ簡易性 | ★★★ | ★★ | ★ |
| スケーラビリティ | ★★ | ★★★ | ★ |

### 推奨

1. **現在の crewvia Worker（小〜中タスク中心）**: **Strategy A を採用し、M6 PreCompact hook を M6 ミッションで実装して自動コンパクション管理を追加する**。

2. **コンパクション管理の優先度**: 今回のベンチマークではコンパクション未発生だったが、大タスクや長期ミッションでは必発。M6 フックの実装が Strategy A を長期運用可能にする鍵。

3. **ハイブリッド閾値**: コンテキスト蓄積量 > 100K tokens で `/clear`（Strategy B）に切り替え、> 500K tokens でプロセス再起動（Strategy C）のフォールバックを設ける。

4. **Strategy C は現状非推奨**: 5h 窓での完走数が A 比 -40%（377 → 228 タスク）であり、小タスク環境での採用コストが高い。大タスク・コンテキスト崩壊対策としてのみ位置づける。
