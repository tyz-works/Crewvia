## 教訓セクション (Phase E → Troi が README/CLAUDE.md に反映)

### L1: Vercel 本番 env は .env.local と独立
**現象**: `vercel env ls` で env var が「存在する (Encrypted)」と表示されていたが、値が空だった。
**学習**: env var の「存在」と「値の非空」は別物。`vercel env ls` は存在確認のみ。値の空非空は `vercel env pull` または実際の動作テストで確認する必要がある。
**チェック方法**: デプロイ後に直接 ntfy テスト通知を送り、iPhone 着信を目視確認する。

### L2: ntfy.ts のサイレントフェイルは診断を困難にする
**現象**: `if (!ntfyUrl || !topic) return;` と `.catch(() => {})` の組み合わせで通知未送信がログに残らない。
**学習**: ミッションクリティカルな通知ロジックは失敗を記録・通知すること。エラーを飲み込むより Redis に `ntfy_send_status` を記録するか、Vercel observability で catch エラーを trace する。

### L3: approval_token は publishApprovalRequest 内でのみ生成される
**現象**: ntfy 通知が届かない場合、token を外部から取得する手段がない (response に含まれない)。
**学習**: デバッグ時は Upstash Redis SCAN で `approval_token:*` を探し、`request_id` フィールドで紐付けできる。harness の `--debug` オプションとして実装を検討。
