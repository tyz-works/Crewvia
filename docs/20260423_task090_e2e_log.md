## 発見事項・教訓

### L4: Preview SSO が自動テストの障壁になる (task_089 L1 の派生)

- **事象**: Vercel Preview が Team SSO 保護下で、Beverly ハーネスから POST 不可
- **対策 (今回採用)**: localhost pnpm dev + 共有 Upstash Redis
- **Phase F 引き継ぎ**: E2E は `pnpm dev` ローカルを推奨。Preview SSO bypass には `vercel@52.0.0` upgrade or `VERCEL_AUTOMATION_BYPASS_SECRET` 設定が必要
- **Worf テスト改善案**: WARN 1-b/1-c/2-d/3-c (open mode で 401 未テスト) → 本番 URL での追加確認を Phase F docs に記載

### L5: vercel curl は POST 非対応 (v50.37.3)

- `-X`, `--request` フラグを受け付けない
- `vercel@52.0.0` upgrade で改善可能性あり

### L6: approval card 作成が Board 表示の前提 (真因)

- **事象**: `verification:*` だけ書いても Board に card が現れない
- **真因**: Board は `approval:index` → `approval:{id}` を読み、card の `task_id` で `verification:{task_id}` を紐付ける
- **対策 (実装済)**: e2e_harness_task090.sh の 2段階フロー (create_card → post_verification)
- **Phase F 引き継ぎ**: crewvia 側 taskvia-verification-sync.sh が approval card なしの task_id に verification を書いている場合、Board に表示されない。crewvia 側でも POST /api/request → task_id 発行の流れを確認すること

### L7: approval card TTL 600s タイミング問題

- **事象**: create_card → post_verification の間に時間が空くと card 消滅 → orphan index
- **対策**: 連続投入戦略 (5s 以内に全状態を投入)
- **根本対策候補**: TTL を延長 (APPROVAL_TOKEN_TTL_SECONDS env) or verification TTL を card TTL と同期

### L8: feature flag は NEXT_PUBLIC_ ではなくサーバーサイド `CREWVIA_VERIFICATION_UI`

- `process.env.CREWVIA_VERIFICATION_UI` (Server Component / Server Action で読む)
- `NEXT_PUBLIC_*` プレフィックスは不要・誤り
- E2E シナリオ手順書の feature flag 変数名を Phase F で修正すること
